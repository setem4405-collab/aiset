# AISET Support

## Purpose

This document explains how to request help, report problems, propose changes, and raise security concerns related to the AISET Program.

AISET is currently in its initial foundation stage. Support is primarily provided through the public GitHub repository and its issue templates.

## Before Requesting Support

Before opening an issue:

- review the repository `README.md`;
- review the relevant publication, specification, schema, registry record, or implementation document;
- search existing issues for similar questions or reports;
- verify the affected version, file path, release tag, or commit;
- collect relevant evidence, logs, screenshots, validation output, or examples;
- remove passwords, private keys, access tokens, personal data, and other sensitive information.

## Documentation Questions

For questions about unclear, incomplete, inconsistent, or incorrect documentation, use the **Documentation issue** template.

Suitable topics include:

- unclear wording;
- broken links;
- missing explanations;
- terminology inconsistencies;
- citation problems;
- formatting issues;
- outdated information;
- possible editorial errors.

## Bug Reports

For reproducible problems in AISET materials, schemas, tools, release artifacts, or future implementations, use the **Bug report** template.

A useful bug report should identify:

- the affected area;
- the affected version or commit;
- steps to reproduce;
- expected behavior;
- actual behavior;
- practical impact;
- relevant environment details;
- supporting evidence.

## Proposals

For architectural, normative, governance, registry, schema, terminology, process, or implementation changes, use the **Proposal** template.

A proposal should explain:

- the problem;
- the proposed change;
- affected scope;
- normative impact;
- compatibility impact;
- interoperability impact;
- security and privacy considerations;
- alternatives considered;
- validation approach.

## Security Issues

Do not disclose exploitable security vulnerabilities in a public issue.

Review:

- [AISET Security Policy](SECURITY.md)

Until a dedicated private reporting channel is established, create only a minimal public notice stating that a potential security issue exists and request a private contact method. Do not include exploit details, credentials, sensitive data, or proof-of-concept code in public.

## Contribution Questions

For questions about submitting changes, review:

- [Contribution Guidelines](CONTRIBUTING.md)
- [Code of Conduct](CODE_OF_CONDUCT.md)
- [Pull Request Template](.github/PULL_REQUEST_TEMPLATE.md)

Substantial changes should normally begin with a focused issue or proposal before a pull request is prepared.

## Publication Questions

For questions about published AISET materials, review:

- [AISET Publications](publications/README.md)
- [DISCOVERY-0001](publications/DISCOVERY-0001/README.md)
- [AISET Registry](registry/README.md)
- [AISET Governance](governance/README.md)

Released publications and release artifacts must not be silently modified.

## Implementation Support

No production AISET software implementation has yet been released.

Future implementation-specific support information should identify:

- supported versions;
- installation requirements;
- known limitations;
- test procedures;
- dependency requirements;
- software license;
- security reporting instructions.

Implementation-specific support documentation may take precedence for the applicable component.

## Response Expectations

The AISET Program is currently maintained during its initial foundation stage.

There is no guaranteed support response time.

Requests may be prioritized based on:

- security impact;
- publication integrity;
- normative impact;
- compatibility or interoperability impact;
- reproducibility;
- completeness of the report;
- relevance to the current AISET roadmap.

## Unsupported Requests

The following may be closed or redirected:

- duplicate issues;
- reports without enough information to evaluate;
- unrelated product support requests;
- requests involving systems not controlled by the AISET Program;
- public disclosure of sensitive vulnerability details;
- personal disputes unrelated to AISET technical or governance work;
- requests to silently alter released publication artifacts;
- requests that conflict with repository licenses or policies.
